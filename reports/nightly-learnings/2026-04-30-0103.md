## Promoted insights

- Insight: After rebasing a fork onto a fast-moving upstream, conflict markers and targeted tests are not enough. Replayed stale fork blocks can leave duplicate declarations, missing imports/helpers, and stale type contracts that only `bun run typecheck` / the actual CI pre-test gates catch.
- Evidence: session `20260430_002927_3f512753`; GitHub job `73730618151` failed in the pre-test gate at `bun run typecheck` after local targeted tests had passed.
- Landing spot: `digests/nightly-learnings-patterns`.
- Why durable: applies to every RBrain/GBrain upstream sync and other active forks.

### Fork rebases must audit historical revert commits as semantic conflicts
- Insight: Historical fork commits that revert upstream PRs cannot be blindly replayed. If those upstream PRs are now legitimate in upstream `master`, the old revert commits must be dropped or consciously rewritten; otherwise the rebase silently removes newly accepted upstream work.
- Evidence: session `20260430_002927_3f512753`; successful sync explicitly dropped old reverts for upstream PRs #460/#461/#464/#465.
- Landing spot: `digests/nightly-learnings-patterns`.
- Why durable: fork syncs often contain policy/revert commits that change meaning as upstream evolves.
