## Mirror

| Source idea | Mirror for Ramy |
|---|---|
| Good strategy has a kernel: diagnosis, guiding policy, coherent action. | For AI Concierge reliability, the mirror is to avoid broad "make it reliable" language. Diagnosis: which failure class breaks first-use trust? Guiding policy: what class of fixes has priority? Coherent actions: which engineering/product/support changes move together? |
| Bad strategy mistakes goals for strategy and avoids the hard challenge. | "Ship Concierge reliability" is a goal, not a strategy. The strategy starts only when the team names the hardest constraint: identity matching, data-sharing policy, evidence display, model behavior, workflow ownership, or support escalation. |
| Chain-link systems are only as good as the weakest link. | Concierge is a chain-link system. Better model prompts do not compensate for bad patient identity, missing audit logs, tenant-boundary ambiguity, or a UI that adds manual work. |
| Focus means coordinated policies plus the right target. | Ramy's current target should remain narrow enough that product, engineering, QA, data, support, and operations can all coordinate around the same reliability outcome. |
| Proximate objectives matter more as complexity rises. | In HIPAA-regulated senior care, distant platform vision should be translated into near objectives: correct identity matching for known cases, auditable recommendation traces, facility-scoped visibility, and no extra manual documentation. |
| Rumelt recommends create-destroy thinking to test proposed strategies. | Ramy's write-first pattern should include an adversarial pass: what would make this reliability plan fail, and which facility staff member still has to compensate manually? |
